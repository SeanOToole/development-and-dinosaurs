Connascence of Name

We're going to start at the bottom of the connascence pile with connascence of name. Connascence of name is a static connascence. A connascence is described as static if it can be discovered by a visual examination of the code as written, as opposed to dynamic connascences that can only be discovered at runtime.

Connascence of name means that multiple components within a system must agree on the name of a particular entity- these components are then coupled together based on that name. Consider the following classes as an example: 


                    class Dinosaur {

                      fun feed() {
                        // Eat
                      }

                    }
                

                    class DinosaurHandler {

                      fun feedDinosaur(val dinosaur: Dinosaur) {
                        dinosaur.feed()
                      }

                    }

So for now let's ignore the design patterns (or lack thereof) or what the dinosaur is actually eating in this scenario. Focussing on the connascence, we have an example of connascence of name here- in both the name of the Dinosaur class and the feed function. Both the Dinosaur class and the DinosaurHandler class need to agree upon the name of the Dinosaur class and the feed function. If you change the name of the Dinosaur class to Chicken, the DinosaurHandler isn't going to work any more unless you change it on that side too. The same goes for the feed function- change it to devour or anything else and we're not going to be feeding anything with that DinosaurHandler without making the same change.

Ok... so what? Well, in this case nothing. Connascence of name is definitely connascence, in that it definitely describes a relationship between different software components that must change at the same time in order to preserve correctness. But... that's the way software works at a fundamental level- it's just how we refer to different components and entities. That's why connascence of name is the weakest level of connascence- if every coupling between components in our codebase can be reduced to a dependency on a name, then we're not in a bad place.

One thing to keep in mind when thinking about connascence of name is that if you want to change the name of something in the future, you're going to have to change that name everywhere that it is referenced. Not a huge problem in statically typed compiled languages like Kotlin, as the code won't successfully compile until you fix the references to the old name- but it will mean potentially larger changes as the codebase grows.
